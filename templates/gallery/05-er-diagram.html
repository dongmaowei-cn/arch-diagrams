用表格化实体 + crow's foot 基数展示数据模型。每个实体就是一张表(标题 / 字段列 / 类型列 / 约束列),关系连线两端用乌鸦脚标记基数(0/1/many),PK 加粗 + 钥匙图标、FK 加 fk 标记 + 虚线箭头指向引用主键。不要在 ER 图里塞业务规则 — 那是 03 状态机 / 01 流程图的事;ER 图只回答"有哪些表、表间怎么连"。

怎么画 · How to draw

ER 图的骨架

  1. 圈出核心域：先列业务名词（订单 / 商品 / 用户 / 支付 …），每个名词是一个候选实体。再判断哪些是 强实体（独立 PK），哪些是 弱实体（依赖父）。
  2. 选 PK：每个实体必有一个 PK · 优先用 bigint id 自增 或 uuid v7。复合 PK 仅用于 junction（双 FK 组合）。
  3. 画 FK 引用：父表 PK ⇐ 子表 FK · 命名 {parent_name}_id · 在子表用 FK 紫色 tag 标注 · 同时给 FK 建索引。
  4. 识别关系类型：1—1（unique FK）/ 1—N（FK 放 N 端）/ M—N（必拆 junction）/ 自引用（FK 引用本表 · 必须 nullable）。
  5. 画 crow's foot 端点：one = 双竖线 / many = 三叉 / 0..1 = 圆 + 竖线 / 0..* = 圆 + 三叉。基数同时决定 FK 是否 nullable。
  6. 标关系动词：用 places / contains / sold as / parent 等动词描述关系 · 不是 has / belongs to · 关系名用白底 badge 浮在连线上。
  7. 加约束标记：UQ（unique · olive）/ IDX（index · teal）/ NN（not null · clay）— 这是本图谱的 [ex] 扩展约定，让常用约束在 ER 图上一眼可见。
  8. 验证范式：1NF 字段原子 · 2NF 非键完全依赖主键 · 3NF 消除传递依赖 · BCNF 进一步严格。常用到 3NF；商业 OLTP 在性能场景下故意反范式（如 order_items.price 快照）。
