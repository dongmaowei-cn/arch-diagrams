01

业务流程图Flowchart · ISO 5807

→ 看完整范本

用 9 种标准节点形状(起止 / 处理 / 判断 / I/O / 数据库 / 子流程 / 文档 / 连接符 / 注释)和 4 种线条(默认 / YES / NO / DB)把一个业务流程从触发到终态画出来。每条规则源自 ISO 5807 国际流程图标准。每张流程图都有且只有一个 START、一个或多个 END,每条边走直角(orthogonal routing)。

元素图鉴 · Element Gallery

10 张卡 · 9 标准节点形状 + 1 边类型集 · 主图实际使用挂 olive ✓ 角标

怎么画 · How to draw

业务流程图的六步骨架

  1. 定义起止:每张流程图都有且仅有一个 START 与至少一个 END。用胶囊形(rounded rectangle)表示触发事件与终态 — 成功用 olive、失败用 rust、中性用 gray 三色区分。
  2. 列出处理动作:把过程中的「动词」抽出来 — 校验、计算、调用、写库、通知。每个动词用一个矩形,标题写动作名,副标写 SLA / 性能指标 / 关键参数。
  3. 标记分支条件:用菱形表示二元判断,出边必须标 YES (olive) / NO (rust) 或具体条件名(如 PASS / FAIL / 等级)。一个菱形只表达一个判断,不要嵌套。
  4. 区分输入输出:用户输入、外部接口回调、消息接收用平行四边形(I/O);持久化存储用圆柱(datastore),读用单向 clay 虚线、写用双向。
  5. 抽出子流程:复杂的子领域(如人工合规审核)用双竖线矩形(subprocess)折叠,避免主图爆炸 — 子流程可以单独画一张图展开。
  6. 处理跨区域跳转:长距离失败路径用连接符(小圆圈 A/B)替代曲线,保持线条永远正交,从而让整图节奏整齐。注释挂方括号引线,不参与流转。

02

时序图Sequence Diagram · UML 2.5

→ 看完整范本

用 参与者 + 生命线 + 激活框 + 消息四件套描述多角色之间按时间先后的调用关系。三种消息线(sync 实线·async 虚线·return 开口)、三种组合片段(alt/loop/opt)、note 折角注释、lifeline 末端 ✕ destroy。每根激活框宽 10px 居中于 lifeline,消息从激活框边缘出发,调用方激活长度 ≥ 被调用方 — 这是 UML 2.5 的硬约束,不要把消息画到 lifeline 中心。

10 张卡 · 主图实际使用挂 olive ✓ 角标

怎么画 · How to draw

时序图的七步骨架

  1. 列出参与者：左到右按发起顺序排 — User → Edge → Internal → External。每个 actor 顶部一矩形 + 下方一根虚线 lifeline。
  2. 切阶段（phase）：用大虚线框把每个业务阶段框起来，左上角标 PHASE n · 名称。阶段是时序图的章节。
  3. 画激活框：每个 actor 在它"正在处理"期间画一根窄竖矩形（宽 10px）覆盖 lifeline。调用方激活长度 ≥ 被调用方 — 调用方必须先激活、后取消激活。
  4. 消息线落在边缘：sync = 实线 + 实心箭头；async = 虚线 + 实心箭头；return = 虚线 + 开口箭头。从一根激活框的边出发，到另一根激活框对应边结束。
  5. self-message 走 U 形：从激活框右边出 → 横走 → 下折 → 横回到激活框右边。
  6. 组合片段 fragment：alt（互斥分支）/ loop（重复）/ opt（条件）— 左上角小斜角标 + 内部虚线 divider 分隔多分支，guard 写在方括号里。
  7. 补语义：折角矩形 = note 注释；lifeline 末端的 ✕ = destroy（连接关闭）；圆圈序号 ① ② … = 强调时间顺序。

03

状态机State Machine · UML 2.5

→ 看完整范本

描述单个对象在不同状态之间的转移 — 每个状态由 guard 条件 + 触发事件 + 副作用 三元组守护。三类终态(成功 olive / 失败 rust / 中性 gray)、初始/终结圆点、决策菱形、自环、内部行为。同一态只能因事件离开,不能因时间被动跳走;每条 transition 必须有触发器,不要画"自动跳转"。

元素图鉴 · Element Gallery

10 张卡 · 主图实际使用挂 olive ✓ 角标

怎么画 · How to draw

状态机的骨架

  1. 定义起止：每张图有一个 initial（实心圆），可以有多个 final（同心圆）。异常退出用 terminate ×。
  2. 抽出主干状态：每个状态用 名词 命名（PAID, SHIPPED），不是动词。简单 → simple state；含子状态 → composite + name-tag。
  3. title + 横线 + actions：状态框分两段 — 标题（名词）与横线之下的 entry / do / exit / defer / 内部 transition。
  4. 三段式 transition：trigger [guard] / action。trigger 必填；guard 与 action 可选。completion transition 是无标签的（子状态自动退出）。
  5. 时间与变化事件：用 after(d) 表示超时（不要写 timeout）；用 when(c) 表示条件变化触发。
  6. 组合状态 + history：用虚线大框 + 左上 name-tag 标记 composite；内嵌 H（回当前层）或 H*（递归回最深层）。
  7. 判断分流：junction（实心小圆 · 编译期静态分流）vs choice（空心小菱形 · 运行时动态分流）；二者形状不同。
  8. 非 UML 扩展打 [ex]：业务专属字段（SLA、retry-counter、priority）属于扩展，挂金色 [ex] 小角标标注 — 让阅读者一眼分清"标准 UML"还是"我们自己加的"。

系统架构图System Architecture · C4 衍生

→ 看完整范本

展示整个系统的分层物理拓扑 — client / edge net / bff / 业务服务 / 中间件 / 数据 / 基础设施。七层 tier-band 横向带状切分,每条边用颜色 + 线型表达调用语义:同步 RPC(实线 plum)、异步事件(虚线 olive)、CDC(波浪 gold)、cache(短虚线 clay)、db(箭头 clay)、抓取(点线 teal)、骨干(粗线 slate)。不要在一张架构图里画时序,时间维度的事情交给 02 时序图。

怎么画 · How to draw

系统架构图的骨架

  1. 定层数：典型 5-7 层。前端 / 接入 / BFF / 业务 / 中间件 / 数据，必要时把横切层（K8s/监控/网格）单独成层。
  2. 定方向：用户请求自顶向下，顶 = 用户最近、底 = 数据最近。横向才是同层服务之间的协作。
  3. 每层一个 dashed 容器框，左侧统一标层号 + 中英文名 + 一句描述，让读者一眼分得清"我在第几层"。
  4. 节点信息四件套：中文名 / 英文技术名 / 技术栈 / SLA 指标。没有数字的节点不算合格。
  5. 主路径高亮：选一条最典型的请求（如下单），用加粗 + 区别色把它从顶串到底，让读者知道"系统的脊柱"在哪。
  6. 区分边类型：同步 RPC、异步事件、缓存读、DB 写、CDC、监控采集 — 5-6 种边类型足够，用颜色 + 虚实区分，并写在图例里。
  7. 把横切点单独画：K8s / Istio / Prometheus / Jaeger 不属于任何业务层，单独一层贯穿是最干净的画法，别在每个节点里画"还有监控"。
  8. 避免画成"框堆"：节点过多时同领域分组 + 行标签（本图 4 层用"交易/商品/边缘" 3 行），比扁平堆叠更易读。

05

ER 图Entity-Relationship · IE 1.1 / Crow's Foot

→ 看完整范本

用表格化实体 + crow's foot 基数展示数据模型。每个实体就是一张表(标题 / 字段列 / 类型列 / 约束列),关系连线两端用乌鸦脚标记基数(0/1/many),PK 加粗 + 钥匙图标、FK 加 fk 标记 + 虚线箭头指向引用主键。不要在 ER 图里塞业务规则 — 那是 03 状态机 / 01 流程图的事;ER 图只回答"有哪些表、表间怎么连"。

怎么画 · How to draw

ER 图的骨架

  1. 圈出核心域：先列业务名词（订单 / 商品 / 用户 / 支付 …），每个名词是一个候选实体。再判断哪些是 强实体（独立 PK），哪些是 弱实体（依赖父）。
  2. 选 PK：每个实体必有一个 PK · 优先用 bigint id 自增 或 uuid v7。复合 PK 仅用于 junction（双 FK 组合）。
  3. 画 FK 引用：父表 PK ⇐ 子表 FK · 命名 {parent_name}_id · 在子表用 FK 紫色 tag 标注 · 同时给 FK 建索引。
  4. 识别关系类型：1—1（unique FK）/ 1—N（FK 放 N 端）/ M—N（必拆 junction）/ 自引用（FK 引用本表 · 必须 nullable）。
  5. 画 crow's foot 端点：one = 双竖线 / many = 三叉 / 0..1 = 圆 + 竖线 / 0..* = 圆 + 三叉。基数同时决定 FK 是否 nullable。
  6. 标关系动词：用 places / contains / sold as / parent 等动词描述关系 · 不是 has / belongs to · 关系名用白底 badge 浮在连线上。
  7. 加约束标记：UQ（unique · olive）/ IDX（index · teal）/ NN（not null · clay）— 这是本图谱的 [ex] 扩展约定，让常用约束在 ER 图上一眼可见。
  8. 验证范式：1NF 字段原子 · 2NF 非键完全依赖主键 · 3NF 消除传递依赖 · BCNF 进一步严格。常用到 3NF；商业 OLTP 在性能场景下故意反范式（如 order_items.price 快照）。

怎么画 · How to draw

泳道图 · 水平的骨架

  1. 定义 Pool：每个参与方（一个公司/系统）画一个 pool。同 pool 内的部门用 lane 细分；不同公司必须是不同 pool。
  2. 抽出 Lane：每个职能/部门 = 一条 lane。常见错：把"用户"放在 pool 边缘（应为独立 pool 才对，但简化主图常合并）。
  3. 找 start / end：每个 pool 至少一个 start（单细圆）+ 一个 end（粗双线圆）。多 end 表多种结果。
  4. 任务用 task 类型：User · Service · Send · Manual · Script 各有图标 — 不要全画 Service 当万能盒子。
  5. 分支用 gateway：XOR ×（if/else，只走一条）· AND +（同时全走）· OR ○+◇（满足条件的都走）。
  6. 连线选对：同 pool 内 = sequence flow（实线）· 跨 pool = message flow（虚线 + 圆头）。不要混用。
  7. 数据用 data object / store：临时单据 = document 形状 · 持久库 = cylinder。用关联虚线连，不是 sequence flow。
  8. 非标记 [ex]：SLA / 超时阈值 / 异步并行标记是业务扩展，挂金色 [ex] 角标 — 与标准 BPMN 区分开。

怎么画 · How to draw

泳道图 · 垂直的骨架

  1. 选方向：phase 横还是纵？此版 phase 纵向 (顶到底 = 时间)，lane 横向 (左到右 = 职能)。适合时间推进强、参与方少 (≤4) 的流程。
  2. 拉 phase column：左侧 60-80px 留给 phase 编号 + 名字 · phase 名 vertical-rl 竖排 · letter-spacing 加大呼吸感。
  3. 划 lanes：每 lane 一列 · 顶部 lane header 带 3px lane 色顶边 · 与下属节点同色形成视觉锚定。
  4. 节点按 lane 着色：customer = external gray · frontend = blue · service = olive · database = plum · 填充 8-10% · 描边同色 1.5px。
  5. 决策菱形特殊：用 clay 虚线菱形 与普通方形节点拉开 · YES 走 olive guard 边 · NO 走 rust fail 边。
  6. 边按类型选：spine olive 2.6px = happy path · slate 1.4px = 次要分支 · rust dashed = 失败 · plum dashed + 空心圆起点 = message flow (跨业务参与方)。
  7. phase divider：相邻 phase 之间画虚线 4 4 dash + 中点 clay ↓ · 强化时间推进感 · 不打断 lane divider。
  8. 失败终态：用 rust 粗双线圆 (stroke-width 2.5) + 圆内写状态码 (400 / 500) · 与 olive 起点细圆形成对仗。

怎么画 · How to draw

微服务架构图的骨架

  1. 切 Tier：client / edge / BFF / core / data 5 层是工业标准 — 横向带 + tier label。
  2. 定核心服务：DDD 限界上下文 → 一个服务一个 box，标 名称 + 技术栈 + QPS + 实例数。
  3. 侧挂 Sidecar：每个服务右上角小 envoy 标记，表示 service mesh 接管流量。
  4. 分线类型：spine（olive 主路径）· sync RPC（橙）· async event（紫虚线）· DB R/W（橙点线）· cache（金虚线）· discover/config（灰/青点线）。
  5. 独立 Platform 列：注册中心 / 配置中心 / Kafka / Saga / 熔断 / 追踪 / 监控 都放右侧独立栏，避免穿插业务服务。
  6. 挂 CORE 徽章：主路径上的关键服务（API Gateway · order-service）挂 olive 实心 CORE 徽章。
  7. 不画太细：物理 pod / 节点 / IP 不画 — 那是部署图。这张是架构图，逻辑视图为主。
  8. 横切 [ex] 角标：熔断 / 限流 / 追踪 / 监控不是 12-factor 标准元素，是工程实践扩展 — 挂金色 [ex] 角标区分。
